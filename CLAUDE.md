# Neuravex Agent & Backend Workflow Guide

> **Notice to AI Agents (Claude & others):** Read this document to understand the dual-backend architecture, workflows, and deployment context for Neuravex.

## 🏗️ Architecture Overview

Neuravex is a React frontend application built with Vite and TailwindCSS that integrates heavily with AI (Gemini) and Supabase. The most critical architectural quirk is its **Dual-Backend System** for processing events:

1. **Local Development (Python FastAPI)**
   - Located in `backend/server.py`
   - Runs locally on `http://localhost:8000` (`uvicorn server:app --port 8000 --reload`)
   - The Vite frontend (`vite.config.js`) proxies `/api/*` requests to this Python server during local development.

2. **Production Deployment (Vercel Serverless Functions)**
   - Located in the `api/` folder (e.g., `api/process-event.js`)
   - Ran via Vercel's Node.js serverless environment. 
   - Uses `vercel.json` to route `/api/(.*)` requests directly to the `api/` functions, while React handles all `/(.*)` frontend routes.

**Important Note for Developers:** If you need to change how an event is processed, you must update **BOTH** `backend/server.py` and `api/process-event.js` to keep local development and production environments perfectly synced.

## ⚙️ How Event Extraction Works (`/api/process-event`)

Both backend implementations expose the exact same API signature and perform the same core workflow.

### 1. Payload Reception
The endpoint `/api/process-event` accepts two types of POST requests:
- **`multipart/form-data`**: Uploading an image or document (PDF) alongside optional text.
- **`application/json`**: Simple text-based requests containing `{ "text": "..." }`.

### 2. Gemini AI Processing
The backend passes the file payload / text contents to **Google Gemini (`gemini-2.5-flash`)**. 
The system uses a highly structured `EXTRACTION_PROMPT` designed to return a standardized JSON schema representing an actionable task/event.

*The AI extracts:*
- `title`, `category`, `venue`, `event_datetime` 
- **Analytical Metrics**: `severity_level` (low/medium/high/critical), `complexity_score` (1-10), `estimated_prep_hours`
- **Actionable Info**: `key_topics` (array of concepts to study), `action_items` (array of task steps)

### 3. Supabase Insertion
Once a valid JSON context is generated by Gemini, the system inserts the event directly into a **Supabase PostgreSQL database** table named `events`.
It automatically adds default fallbacks (like `status="pending"`) if Gemini omits them to ensure database constraint compliance.

## 🔑 Environment Variables Required

You must provide the following in your `.env` file at the root:

```env
GEMINI_API_KEY=your_gemini_api_key
SUPABASE_URL=your_supabase_project_url
SUPABASE_ANON_KEY=your_supabase_anon_key
```

Both the Vercel Node environment and Python FastAPI load these exact variables.

## 🚀 Running the App Locally

To spin the application up properly:

1. **Start the Python Backend (Terminal 1)**
   ```bash
   cd backend
   pip install -r requirements.txt
   uvicorn server:app --port 8000 --reload
   ```

2. **Start the Frontend (Terminal 2)**
   ```bash
   npm install
   npm run dev
   ```

3. Ensure that any request from the React app (on Vite's port `5173`) going to `/api/` successfully proxies to `8000`.

## 🛠️ Future Maintenance / Scaling Notes

- **To modify the DB schema:** If new columns are added to the Supabase `events` table, you need to update the Gemini `EXTRACTION_PROMPT` in both backends to capture the new keys, and whitelist them in the backend insertion logic loop.
- **Handling Formidable in Next.js/Vercel:** Vercel serverless requires disabling standard body parsers `export const config = { api: { bodyParser: false } };` so that `formidable` can parse the incoming raw multipart chunks (seen in `process-event.js`). Do not remove this config.
